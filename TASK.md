предполагаемая архитектура:
делаем сервис с чётким разделением ответственности — LLM только понимает запрос, а все вычисления выполняются детерминированно на стороне Python (через pandas). CSV в модель не отдаём.

1. сначала разобраться со структурой данных (`data.csv`) и требованиями из `TEST.md`
   — какие поля есть, какие обязательны, какие нужно вычислять

2. подготовить слой данных:

   * загрузчик CSV из `data/`
   * валидация обязательных колонок
   * нормализация timestamp
   * расчёт производных полей:

     * `horizontal_speed`
     * `acceleration`
     * `datetime_moscow`
     * `hour_moscow`

3. API слой (FastAPI):

   * `POST /query` — основной эндпоинт для обработки запросов

4. разделение логики:

   * LLM используется как классификатор (intent + параметры)
   * дальше вызываются pandas-обработчики, которые считают результат без участия модели

5. поддержка основных типов запросов:

   * агрегации (max / min / avg скорости)
   * фильтрация по качеству позиционирования
   * временные срезы
   * аналитика (резкое торможение / ускорение)
   * геопространственный фильтр (например, попадание в регион M11)

6. отдельно продумать гео-логику
   — как определять принадлежность точки региону (скорее всего через bounding box или shapely)

7. упаковка:

   * Dockerfile
   * docker-compose для запуска

8. тестирование:

   * покрыть ключевые вычисления (скорость, ускорение, фильтры)
   * проверить корректность обработки intent’ов

итоговая идея:
LLM — это «парсер запроса», который возвращает что считать, а не считает сам. Всё, что можно посчитать детерминированно — считаем в коде, чтобы система была предсказуемой и тестируемой.
